# 架构与行为契约

这是“我想听歌！”的唯一技术事实来源。`README.md` 面向使用者，`DEVELOPMENT.md` 用于定位模块和改动流程；当它们与本文件或测试冲突时，以本文件和测试为准。

## 定位与边界

插件只做三件事：把一首可信的歌作为语音发送、把用户明确要求下载的歌作为文件发送、在用户明确说“搜歌/找歌”时展示候选。它不是通用播放器框架。

- Bilibili 是唯一的搜索、解析和播放源。
- 平台搜索保持清理后的用户关键词，不自动追加“原版”。结构化条件的首轮结果不足十条时，才额外以仅歌名补充一次候选。
- Live、翻唱、AI、DJ、Remix、伴奏、变速、片段、剪辑和 `MV` 等标题标记都是候选证据，不是本地硬过滤条件；Bilibili 分类不参与过滤或重排，版本选择由用户意图和 LLM 完成。
- 用户原文中的 AV/BV 或普通 Bilibili 视频链接可精确定位视频详情，但下载仍需选择具体分 P。
- 没有指定曲目的“唱首歌”是推荐一首合适的具体歌曲，不是回复点歌说明。
- 不实现歌词、评论、封面、歌单、收藏、播放历史、音质选择或平台选择。
- 候选列表最多十条。下载必须由用户从当前候选中确认，不能自动选歌后直接发送文件。
- 不为了未来扩展添加音源注册表、`BaseSource`、播放器继承树或第二音源。这些抽象会扩大行为面并允许错误地换歌。

## 分层与依赖方向

```text
AstrBot / WebUI
    main.py
       |-- SearchService / DeliveryService ---- core/services.py
       |       |-- snapshot store ------------ core/selection.py
       |       |-- delivery guard ------------ core/matcher.py
       |       |-- Bilibili protocol --------- core/bilibili.py
       |       `-- one-shot local media ------ core/media.py
       `-- administrator credentials -------- core/accounts.py
```

`main.py` 是唯一允许依赖 AstrBot 的层：生命周期、命令、LLM 工具、消息发送、会话等待和 WebUI 路由都在这里。`core/` 不依赖 AstrBot；其中 `matcher.py` 还是纯函数层，禁止 HTTP 和副作用。插件启动时仅创建一个共享 `aiohttp.ClientSession`，将其注入 Bilibili、媒体和账号服务；停止时按反向顺序关闭。

| 模块 | 拥有的职责 | 明确不拥有的职责 |
| --- | --- | --- |
| `core/models.py` | 候选、音频、媒体、快照等小型不可变领域模型 | 协议或聊天行为 |
| `core/matcher.py` | 已展开 Bilibili 分 P 的时长、去重与上限守卫 | HTTP、AstrBot、版本语义判断或自动换歌 |
| `core/bilibili.py` | WBI、搜索、视频详情、DASH、二维码协议 | 聊天意图和候选决策 |
| `core/services.py` | 搜索展开、受限候选集合与已选页的交付编排 | 发送 AstrBot 消息 |
| `core/media.py` | 有界完整交付、备用 URL、无重编码封装、单次临时文件清理 | Bilibili 业务语义 |
| `core/selection.py` | 会话绑定、过期和不可猜测的搜索快照 | 持久化播放历史 |
| `core/accounts.py` | 管理员 Bilibili Cookie 与二维码会话 | 对聊天公开账号信息 |

## 身份、会话与请求流程

### 不可替换的候选身份

`BilibiliCandidate` 的身份是具体 `bvid:cid`，不是视频标题、搜索词或整个视频。关键词搜索先取少量视频，再展开成具体分 P；AV/BV 则先读取指定视频详情，再展开全部可交付分 P。两条路径都按来源顺序经过与当前动作相符的时长、去重和上限守卫后才生成 `SearchSnapshot`。音频解析只能使用这个已经选定的 `bvid:cid`；任何解析失败都必须显式失败，不能重新按标题搜索或改选另一个分 P。

每次搜索都形成与 `event.unified_msg_origin` 绑定、有效期五分钟的短期候选集合。候选身份只在插件内部使用：LLM 只可从本次隐藏集合给出的序号中选择，用户只可从当前显示列表回复序号。隐藏候选授权在每个会话中只有一个，受同一有效期和 1024 条全局上限约束；新消息或新请求会使旧授权失效。用户原文中的 AV/BV 只能作为读取详情的入口，链接、原始 `bvid`、`cid` 和编造的候选标识都不能绕过快照直接交付。所有文件交付都要求“第 N 首 下载”；单独的“第 N 首”默认听歌。

### 直接听歌

```text
用户意图
  -> LLM 理解歌名、歌手和版本偏好
  -> Bilibili 搜索 -> 展开为 bvid:cid -> 语音时长守卫（最多 15 分钟） -> 最多十条隐藏候选
  -> LLM 只在这组候选内作语义判断并选择一项
  -> 终止型交付工具验证候选身份，并行准备媒体与发送简短前言
  -> Record.fromFileSystem()
  -> finally: MediaStore.release() -> 删除本地媒体
```

直接听歌让 LLM 完成两项不同的判断：先将对话收敛为结构化的歌曲条件，再在插件提供的受限候选中判断最合适的一项。候选检索、页级过滤、候选身份验证和媒体交付仍由插件完成。终止型交付成功后返回真正的 `None`，不会再要求 LLM 在媒体之后补一条文本。前言中的歌曲名必须来自已验证候选；可选祝语必须简短，且不能包含“正在找”“检索”“换用”“找到了”等过程性或尾随确认文案。AstrBot 的 `provider_settings.display_reasoning_text` 应保持关闭，避免把模型推理内容显示给聊天用户。

### 下载与显式搜歌

```text
下载 <关键词>、搜索歌曲 <关键词>、AV/BV 或由 LLM 判定为显式搜索
  -> 关键词搜索，或由 AV/BV 读取视频详情
  -> 相同的 SearchService 与页级可交付性守卫（不施加语音时长上限）
  -> 最多展示十个具体页的“标题 + 时长”；超过 15 分钟的项标记“仅可下载”
  -> 插件启动唯一的 SessionWaiter，用户在 90 秒内选择序号
  -> 对“仅可下载”项回复单独序号时，保留原候选等待并提示回复“序号 下载”
  -> 从原快照取回同一 bvid:cid -> 交付
```

下载请求与普通搜歌共享同一选择语法：“序号”听歌，“序号 下载”发送文件。候选展示不显示 UP 主。下载无论条件多明确都不由 LLM 自动选定版本；它必须先进入用户选择路径。命令与 LLM 搜索都调用同一条 `SessionWaiter` 路径；新搜索会安全替换同一会话的旧等待。候选展示和交付之间绝不重新搜索，LLM 也不参与后续选择或发送。真正等待满 90 秒时，插件发送一条本次搜索已结束的提示；同一会话中的任意非选择新消息、新搜索或替换操作都会静默取消旧候选状态。命令选择路径不额外发送“正在处理”或成功确认文本。

### 管理员账号

```text
Dashboard 唯一管理员
  -> /astrbot_plugin_listen_music/accounts 页面
  -> 创建二维码 -> SSE 等待状态变化 -> 原子写入本地 Cookie
  -> BilibiliClient 在请求时取得 Cookie 副本
```

每次仅允许一个活动二维码会话，且会话绑定 Dashboard 管理员。账号 API 只返回账号状态、昵称、二维码位图和运行健康状态；Cookie、二维码轮询令牌和原始二维码 URL 不得进入响应、聊天、LLM 或日志。退出只删除本地凭证，不撤销 Bilibili 远端会话。未登录是允许的降级状态：仍可匿名搜索和尝试播放。

账号页只管理 Bilibili 账号与运行健康状态，不承载聊天策略。候选数固定为十条，不新增账号页设置；将来确有管理员配置需求时，应使用 AstrBot 标准插件配置，而不是在账号页另建持久化设置或 API。

## 候选过滤契约

### 参考来源与独立实现

NeriPlayer 是本项目的历史参考，不是 Kotlin 代码或策略的移植。它帮助说明两个边界：搜索视频和实际可播放分 P 必须区分；确定交付页后不能因解析失败而静默换成其他页或其他音源。它的标题、艺人、时长等多证据思路可用于审阅候选质量，但本项目不复制其评分公式或最低分阈值；受限候选中的语义判断由 LLM 完成。它是 GPL-3.0 项目，不得复制、翻译或改写其代码；完整致谢和链接见 [REFERENCES.md](REFERENCES.md)。

### 当前规则，必须保持可解释

1. LLM 从对话中提供歌名、可选歌手与可选版本偏好。插件组合清理后的关键词请求 Bilibili，但绝不自动追加“原版”“原唱”或其他版本词，以免扭曲平台召回。
2. Bilibili 负责召回。插件保留其结果作为候选输入，不对标题、歌手、上传者或时长计算本地相似度、分数或阈值，也不据此替 LLM 做语义选歌。结构化条件的首轮候选不足十条时，插件只再查询一次仅歌名结果；首轮顺序不变，合并时按 `bvid:cid` 去重。
3. 用户原文包含 AV/BV 或普通视频链接时，插件不进行关键词搜索，而是用 Bilibili 详情接口取得服务端返回的 canonical `bvid` 与分 P。失效引用明确失败，不能退回关键词搜索并命中另一首歌。
4. 搜索或详情结果必须先展开成具体分 P。普通关键词的多 P 视频只有在结构化歌名可确认页面身份时才保留；精确视频引用保留其全部可交付分 P。每个候选固定为 `bvid:cid`，隐藏与显式候选集最多各保留十项。
5. 本地过滤只执行可验证的交付边界：无效时长、重复页和候选数量上限。直听候选额外排除超过 15 分钟的页；显式搜索和下载不设时长上限，超过 15 分钟时仅标记为“仅可下载”。Live、翻唱、AI、DJ/Remix、伴奏、变速、片段、剪辑和 `MV` 等标题标记不会被本地代码解释或排除；Bilibili 分类不参与候选处理。
6. 直听时，LLM 在受限候选内依据用户意图与标题证据选择一项；插件验证其仍属于本次候选集合后才交付。显式搜索展示可交付结果，展示文字仅为标题和时长，不显示 UP 主。下载一律等待用户选择；未标注候选不被宣称为“原版”。

修改查询清理、补召回、分 P 身份判定或可交付性规则前，必须先在 `tests/test_matcher.py` 和相关服务测试中加入正反例。测试至少覆盖版本标签保留、候选去重、首轮顺序、补召回、AV/BV 详情、分页和候选数量边界。修改 LLM 协作路径时，还必须覆盖候选范围验证、直听无尾随文本、原始消息 AV/BV 识别和下载必须人工选择。

## 资源与故障边界

- HTTP 会话连接总数为 `16`，每主机 `8`；搜索视频详情最多并发 `4` 个，媒体从下载到发送完成最多并发 `2` 个。
- 使用自然语言工具调用时，AstrBot 的 `provider_settings.tool_call_timeout` 应至少为 `300` 秒，给直听音频的解析、拉取和封装留出余量。下载不占用 LLM 工具调用：它在用户确认候选后由会话处理器交付。
- 语音交付最多 `15` 分钟、`25 MiB`；一次获取中所有备用 CDN 尝试共用 `180` 秒总超时。
- 显式下载不限制音频时长，单文件最多 `100 MiB`；一次获取中所有备用 CDN 尝试共用 `900` 秒总超时。文件超限时明确失败，不自动降低音质或突破边界。
- 候选时长上限按动作应用：LLM 直听只取得可作为语音发送的候选；显式搜索和下载保留长音频，并在列表中标记“仅可下载”。
- DASH 音频必须用 `ffmpeg` 无重编码封装为 M4A；没有 `ffmpeg` 时账户页显示缺失状态，听歌和下载返回清晰错误。
- 插件启动时，`MediaStore.reclaim_stale()` 在任何交付前清理上次异常退出留下的媒体目录。`MediaStore.prepare()` 为一次交付创建唯一的临时文件，并持有交付名额直到 `_send_delivery()` 的 `finally` 调用 `release()`；慢发送会排队，而不是积累待发送文件。插件停止时取消在途准备并清空媒体目录。没有缓存、共享下载或后台过期任务。
- DASH 封装时，原始音频与 M4A 会短暂同时存在；两个 100 MiB 下载并发时，媒体目录峰值约为 `400 MiB`。这是传输期间的上界，发送后立即回收，不会随运行时间增长。
- `weixin_oc` 适配器的出站链支持 `File` 但忽略 `Record`，因此语音听歌会直接交付同一媒体文件；其他平台仍优先语音，适配器明确报错时再回退文件。
- Bilibili 下载请求使用 Referer、UA 和受控的 Cookie 副本；备用 URL 仅是同一已选音频流的 CDN 备用地址，不能成为换歌机制。
- Cookie 存在插件数据目录的 `accounts.json`，原子写入且权限为 `0600`。不保存播放历史、收藏或平台写操作。

## 修改与验证

先改所属层，不要为了少量重复引入跨层框架。改动交付路径时必须检查：媒体准备失败、前言发送失败、取消、发送失败、临时文件删除以及插件停止。改动登录或 WebUI 时必须检查管理员身份、二维码会话所有权、SSE 脱敏和 Cookie 文件权限。

测试地图：

| 测试 | 保护的契约 |
| --- | --- |
| `test_matcher.py` | 查询清理、按动作应用的时长守卫、去重和候选范围 |
| `test_services.py`、`test_selection.py` | 搜索展开、补召回、精确视频页、长候选标记、快照隔离与过期 |
| `test_media.py` | 语音与下载的独立资源上限、共享备用 URL 超时、完整交付背压、启动回收与临时文件清理 |
| `test_bilibili.py` | WBI、分页、DASH、请求头和 Cookie 使用 |
| `test_accounts.py` | 凭证脱敏、原子写入、二维码会话归属 |
| `test_main.py` | AstrBot 命令、LLM 候选协作、终止型交付、消息和 WebUI 边界 |

```bash
python3 -m unittest discover -s tests -v
python3 -m py_compile main.py core/*.py tests/*.py
```

单元测试不需要真实 AstrBot、ffmpeg 或网络。涉及实际适配器消息发送、二维码 SSE、音频封装和 Bilibili 真实请求的改动，还必须在 AstrBot `>=4.26,<5` 中手工验收。
