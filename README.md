# 我想听歌！

> 面向 AstrBot 的 Bilibili 听歌插件：直接把合适的歌曲作为语音发送；需要比较版本或下载时，先由用户选择，再发送音频文件。

`astrbot_plugin_listen_music` 支持 AstrBot `>=4.26,<5`。音源固定为 Bilibili：插件搜索视频、定位到具体分 P，并只解析该页的音频流。它不内置或托管歌曲，内容可用性受 Bilibili 的公开状态、账号权限、地区和网络环境影响。

## 工作机制

```text
直接听歌
用户意图 -> LLM 提取歌名、歌手和版本偏好 -> Bilibili 候选分 P
         -> LLM 仅在本次候选中选定一首 -> 发送语音

明确搜索或下载
用户请求 -> Bilibili 候选分 P -> 插件展示列表 -> 用户选择 -> 发送语音或文件
```

每个候选都锁定到具体的视频分 P。选定后只解析该页；解析失败会明确告知，不会悄悄替换成另一首歌或另一个分 P。

## 功能与交付边界

| 请求 | 选曲方式 | 交付 | 限制 |
| --- | --- | --- | --- |
| 直接听歌、点歌、`唱首歌` | LLM 在当前 Bilibili 候选中选择 | 语音 | 最长 15 分钟、最大 25 MiB |
| `搜索歌曲 <关键词>` | 用户回复候选序号 | 语音或文件 | 最多显示 10 条 |
| `下载 <关键词>` | 用户回复 `序号 下载` | 文件 | 不限时长、单文件最大 100 MiB |
| AV/BV 号或含 AV/BV 的标准 Bilibili 视频链接 | 用户选择具体分 P | 语音或文件 | 精确读取指定视频，不回退关键词搜索 |

直听时，未指定版本会优先普通完整录音；用户提出的 Live、翻唱、AI、Remix、DJ、伴奏或 MV 等偏好会参与候选判断，而不会被机械过滤。下载始终由用户确认具体版本，LLM 不会自动下载。

候选列表只显示视频标题和时长，不显示 UP 主。超过 15 分钟的候选会标记“仅可下载”。

精确视频定位识别文本中的 AV/BV 号，以及包含该标识的 `bilibili.com/video/...` 链接；`b23.tv` 短链暂不解析。即使链接带有 `?p=` 参数，插件仍会列出全部可交付分 P 供用户选择。

```text
搜索歌曲 晴天
1                 # 听第 1 首
1 下载            # 下载第 1 首
取消              # 结束当前选择
```

## 部署要求

| 项目 | 要求 |
| --- | --- |
| AstrBot | `>=4.26,<5` |
| Python | `>=3.10`；AstrBot 插件数据目录必须可写 |
| 系统工具 | 安装 `ffmpeg`，且 AstrBot 进程可在 `PATH` 中找到它；Docker 部署时须安装在 AstrBot 容器内 |
| 网络 | 可通过 HTTPS 访问 Bilibili 的 API、网页、登录域名和音频 CDN |
| LLM | 直接听歌需要当前模型支持 AstrBot 工具调用；若当前 Provider 提供相关设置，建议工具调用超时不少于 300 秒，并关闭推理过程展示 |
| Python 依赖 | AstrBot 运行环境中的 `aiohttp`；WebUI 扫码登录还需要 `qrcode` 与 Pillow |

插件通过 `ffmpeg` 将 Bilibili DASH 音频无重编码封装为 M4A。缺少 `ffmpeg` 时，管理页会显示健康检查失败，听歌和下载会返回明确错误。

若 AstrBot 的 Python 环境尚未安装二维码依赖，可在该环境中执行：

```bash
python -m pip install "qrcode[pil]"
```

Bilibili 账号不是搜索和基础播放的前提。未登录时仍会匿名搜索和尝试播放；登录可提高部分受限内容的可用性。

自然语言的“听歌”“点歌”“唱首歌”和“下载歌曲”由模型调用插件工具完成。模型不支持或未启用工具调用时，仍可使用 `搜索歌曲 <关键词>` 展示候选，再由用户回复序号听歌或下载。

## 安装与更新

在 AstrBot WebUI 中通过仓库地址安装：

```text
https://github.com/57Darling02/astrbot_plugin_listen_music
```

通过仓库或插件市场安装后，AstrBot 会记录安装来源，WebUI 可以检查更新。已有手动复制目录或 ZIP 安装的实例，需要重新绑定该仓库来源，或按仓库方式重装一次。


## 管理员账号

插件页面位于 AstrBot WebUI 的“我想听歌！”。仅登录用户名与 AstrBot `dashboard.username` 一致的 Dashboard 主账号可以：

- 查看 Bilibili 登录状态和 `ffmpeg` 健康状态。
- 创建、取消 Bilibili 二维码登录。
- 删除本地账号凭证；该操作不会撤销 Bilibili 侧已有会话。

聊天用户、LLM 工具和外部接口不能读取、导入或导出 Cookie。凭证保存在 AstrBot 插件数据目录的 `accounts.json`；媒体文件仅在发送期间写入临时目录，发送完成后立即清理。

登录仅用于访问该账号本来可访问的 Bilibili 内容；插件不绕过会员、地区、版权或平台访问权限。部署者应自行遵守 Bilibili 的服务条款和内容授权要求。

## 运行资源

| 资源 | 上限或策略 |
| --- | --- |
| Bilibili 视频详情请求 | 最多并发 4 个 |
| 音频准备与发送 | 最多并发 2 个 |
| 语音拉取 | 所有备用 CDN 共用 180 秒 |
| 文件拉取 | 所有备用 CDN 共用 900 秒 |
| 搜索候选 | 与会话绑定，5 分钟有效 |
| 用户选歌 | 等待 90 秒 |

插件不保留播放历史、收藏或媒体缓存。DASH 封装期间原始音频和 M4A 会短暂同时存在；两个 100 MiB 文件下载并发时，临时空间通常需要约 400 MiB，宿主机应预留更多余量，发送后会回收。`100 MiB` 是插件上限，适配器自身的文件限制可能更低。

`weixin_oc` 不支持本插件使用的语音消息类型时，会直接收到同一首歌的音频文件。其他适配器优先发送语音，适配器明确报错后才回退为文件。

## 部署后验收

1. 在插件页确认 `ffmpeg` 状态为可用；可选地完成 Bilibili 扫码登录。
2. 发送 `来一首周杰伦的晴天` 或 `唱首歌`，应由 LLM 选定候选并发送语音。
3. 发送 `搜索歌曲 晴天`，应收到最多 10 条“标题 + 时长”候选；回复 `1` 应收到语音，回复 `1 下载` 应收到文件。
4. 发送 `下载 BV1...` 或包含 AV/BV 的标准 Bilibili 视频链接，应收到该视频的分 P 候选，而不是关键词搜索结果。

## 故障排查

| 现象 | 优先检查 |
| --- | --- |
| 管理页提示缺少 ffmpeg | 执行 `ffmpeg -version`，确认可执行文件在 AstrBot 进程的 `PATH` 中 |
| 无法生成登录二维码 | 在 AstrBot 的 Python 环境安装 `qrcode[pil]`，并检查管理员权限与 Bilibili 网络连通性 |
| 搜索正常但无法发送 | 检查 `ffmpeg`、适配器文件大小限制和 Bilibili 音频权限 |
| WebUI 不显示新版本 | 确认从仓库或插件市场安装，且远端 `metadata.yaml.version` 已递增 |
